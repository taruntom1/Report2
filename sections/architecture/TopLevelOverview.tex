\subsection{Top-level Overview}

The proposed LoRa-based Vehicle-to-Vehicle system employs a \textbf{three-layer architecture} designed 
to achieve clear separation of concerns across application functionality, network intelligence, and 
radio transmission. This stratified design facilitates distributed safety message dissemination among 
networked vehicles operating over constrained long-range wide-area network links while sustaining 
both modularity and testability.

The \textbf{Application Layer} serves as the interface through which vehicle services originate and 
consume network messages. This layer is responsible for generating outbound traffic destined for
 neighboring vehicles and processing inbound messages delivered by the network substrate. The 
 Application Layer remains independent of specific message routing policies or hardware characteristics, 
 establishing a clean boundary for service-level logic.

The \textbf{Network Layer} functions as the control core of the system, responsible for managing message 
delivery and forwarding decisions. This layer wraps application payloads with requisite network metadata 
(such as hop counts, timestamps, and originator identifiers), evaluates whether received packets should 
be delivered to the local node and/or forwarded to neighbors, suppresses duplicate transmissions to 
conserve spectrum, and schedules relay timing to avoid contention and collision. By centralizing these 
control policies, the Network Layer provides a unified point for deployment of adaptive forwarding strategies 
and congestion management.

The \textbf{Link Layer} abstracts the radio transmission boundary and provides the physical means for 
frame transmission and reception over long-range wireless hardware. This layer manages packetization of 
serialized messages, executes radio state transitions and channel sensing operations, and implements 
optional mechanisms such as selective acknowledgment retry for reliability. By encapsulating hardware 
specifics, the Link Layer enables the system to support multiple physical backends without requiring changes 
to higher-layer logic.

During standard operation, application payloads traverse the stack downward through successive layers 
for transmission, while received radio frames traverse upward through the Link and Network layers before 
delivery to the Application Layer. To enable both production deployments and simulation-driven testing, 
the Network Layer maintains explicit hardware independence through a well-defined abstract interface with 
the Link Layer. This abstraction permits the system to operate with either material radio devices or 
software-based channel simulators, supporting thorough evaluation and validation in controlled laboratory 
settings. The overall layering and bidirectional data flow are illustrated in Figure~\ref{fig:system_architecture}.
